Java面试题之计算机网络

发表于 2021-04-08 更新于 2021-04-29 分类于 面试题

本篇汇总常见的计算机网络面试题。

1.OSI模型及各层协议

  • 物理层：建立、维护、断开物理连接。

  • 数据链路层：建立逻辑连接，进行硬件地址寻址，差错检验。

    协议：PPP、L2TP

  • 网络层：进行逻辑地址寻址，实现不同网络间的路径选择。

    协议：IP、ICMP、RIP、ARP、OSPF

  • 传输层：定义传输数据的协议端口号，进行流量控制和差错检验。

    协议：TCP、UDP

  • 会话层：建立、管理、终止会话。

  • 表示层：数据的表示、安全、压缩。

  • 应用层：网络服务与最终用户的接口。

    协议：HTTP、HTTPS、FTP、TFTP、DNS、TELNET、SMTP、POP3

2.GET和POST请求的区别

（1）get的url参数可见；post的url参数不可见；

（2）get通过拼接url传递参数；post通过body体传递参数；

（3）get请求可以缓存；post请求不能缓存；

（4）get请求在页面后退时不产生影响；post请求在页面后退时会重新提交请求；

（5）get一般传输数据大小不超过2K-4K；post传输数据大小根据配置文件设定，可以无限大；

（6）get请求产生1个TCP数据包；post请求产生2个TCP数据包。

3.TCP和UDP的区别

TCP（传输控制协议）：是一种面向连接的、可靠的、基于字节流的传输层通信协议。

UDP（用户数据报协议）：是一种无连接的、不可靠的、面向事务的信息传送服务。

（1）连接方面：

  • TCP面向连接。

  • UDP无连接，即发送数据之前不需要建立连接。

（2）安全方面：

  • TCP提供可靠的服务，传送的数据无差错、不丢失、不重复且按序到达。
  • UDP尽最大努力交付，因此不保证可靠。

（3）传输效率：

  • TCP传输效率相对较低。
  • UDP传输效率高，适用于对高速传输和实时性有较高的通信或广播通信。

（4）连接对象数量：

  • TCP连接点对点，一对一。
  • UDP连接支持一对一、一对多、多对一和多对多。

4.HTTP和HTTPS的原理

HTTP：超文本传输协议HyperText Transfer Protocol。

原理：客户端的浏览器首先通过网络与服务器建立连接，该连接通过TCP完成，端口号通常为80。建立连接后，客户端发送一个请求给服务器，请求格式为URL+协议版本号+请求修饰符+客户机信息等。

服务器接到请求后，给予相应响应信息，格式为一个状态行，包括协议版本号+成功或错误代码+服务器信息+实体信息等。

HTTPS：以安全为目标的HTTP通道，是HTTP的安全版。HTTPS的安全基础是SSL。SSL协议位于TCP/IP协议和各种应用层协议之间，为数据通讯提供安全支持。

原理：HTTP直接将报文信息传输给SSL套接字加密，SSL将加密后的报文发送给TCP套接字，TCP套接字将加密后的报文发给目的主机，目的主机通过TCP套接字获取加密后的报文给SSL套接字，SSL解密后交给对应进程。

SSL协议分为SSL记录协议和SSL握手协议。

SSL记录协议为高层协议提供数据封装、加密、压缩等功能。

SSL握手协议建立在SSL记录协议之上，用于实际数据运输开始前的身份验证、交换秘钥等。

5.HTTP和HTTPS的区别

（1）HTTPS协议用到CA申请证书，而HTTP不需要。

（2）HTTP是超文本传输协议，信息明文传输；HTTPS是具有安全性的SSL加密传输协议。

（3）HTTP使用80端口，HTTPS使用443端口。

（4）HTTP连接简单，是无状态的；HTTPS是由SSL+HTTP协议构建的可加密传输、身份验证的网络协议，更加安全。

HTTPS相对于HTTP的改进：增加双向身份认证、使用加密算法保证数据传输安全性、防止报文重放攻击。

6.HTTPS加密过程

（1）HTTPS基于TCP协议，客户端先和服务端建立连接。

（2）服务端把其CA证书返回给客户端，证书包括公钥S.pub、颁发机构及有效信息等。

（3）可以通过浏览器内置的根证书（含C.pub）验证CA证书的合法性。

（4）客户端生成随机对称加密密钥，通过公钥S.pub加密发给服务端。

（5）客户端与服务端通过对称密钥加密数据进行HTTP通信。

7.什么是CA证书

CA证书确保服务端的公钥准确无误，未被修改。

证书通常包括服务端的公钥、发行者的数字签名、证书使用的签名算法、证书发布机构、有效期等信息。

8.加密算法的类型

加密算法分为三大类：单向加密、对称加密和非对称加密。

其中对称加密中，加密和解密使用同一个密钥；

而非对称加密中，使用两个密钥公钥和私钥，公钥加密需要私钥解密，私钥加密需要公钥解密。

9.常见的加密算法

对称加密算法

DES：初代对称加密算法，是数据加密标准。使用javax.crypto包下的DESKeySpec等实现。

AES：使用最多的对称加密算法。

PBE：基于口令的加密算法，是对以上算法的包装升级。

非对称加密算法

RSA：支持变长密钥的公钥算法。

DSA：数字签名算法，是数字签名标准。

ECC：椭圆曲线密码编码学。计算量小，处理速度快，且抗攻击性强，对带宽要求低。

单向加密算法（使用Hash散列）

MD5：单向散列算法，非可逆，相同明文对应相同密文。易受密码分析的攻击。

SHA：可对任意长度的数据运算生成一个160位的数值。运行速度较慢。

6.HTTP常见状态码

状态码 描述
203 非授权信息
301 请求的资源已被永久移动至新URI
304 所请求的资源未修改
305 所请求的资源必须通过代理访问
400 客户端请求语法错误
401 要求用户身份认证
403 服务器拒绝执行该请求
404 服务器无法根据请求找到资源
500 服务器内部错误

11.三次握手

三次握手的过程

起初客户端处于closed的状态，服务端处于listen状态。

第一次握手：客户端给服务器发送一个SYN报文，并指明客户端的初始化序列号ISN(c)。此时客户端处于SYN_SEND状态。

第二次握手：服务器收到客户端的SYN报文后，会以自己的SYN报文作应答，并且指定自己的初始化序列号ISN(s)，同时把客户端的ISN(c) + 1作为ACK的值，表示自己已经收到客户端的SYN。此时服务器处于SYN_RECEIVED状态。

第三次握手：客户端收到SYN报文后，发送一个ACK报文，把ISN(s) + 1作为ACK的值，表示已经收到了服务器的SYN报文。此时客户端处于established状态。

服务器收到ACK报文后，也处于established状态，此时双方建立起连接。

三次握手的作用

（1）确认双方的接收及发送能力是否正常。

（2）指定自己的初始化序列号，为后期可靠传送做准备。

（3）对于HTTPS协议，该过程还会进行数字证书的验证及加密密钥的生成。

为什么不能两次握手

差别在于最后客户端对服务器的确认。防止已经失效的连接报文请求突然传送至服务器，从而产生错误。

例如：客户端发送了第一个请求连接并且没有丢失，只是因为在网络结点中滞留时间较长。此时由于客户端迟迟没有收到ACK确认报文，以为服务器没有收到，会重新发送，但是网络通畅时之前的报文又传送过来了，于是客户端和服务器会建立两次连接，造成不必要的错误和资源浪费。

而若是三次握手，即使失效报文传至服务器，服务器回复ACK报文，客户端也不会再次确认，于是连接不会再次建立。

为什么不能四次握手

因为服务器向客户端发送ACK报文和SYN报文可以优化为一步，也即是三次握手。

12.四次挥手

四次挥手的过程

起初均处于established状态。假设客户端发起关闭请求：

第一次挥手：客户端发送一个FIN报文，报文中指D定一个序列号ISN(c)。此时客户端处于FIN_WAIT_1状态。

第二次挥手：服务器收到FIN后，发送ACK报文，将ISN(c) + 1作为ACK的序列号，表示已收到客户端报文。此时服务器处于CLOSED_WAIT状态。（客户端收到ACK后，进入FIN_WAIT_2状态）

第三次挥手：当服务器处理完数据后，发送FIN报文，指定序列号ISN(s)。此时服务器处于LAST_ACK状态。

第四次挥手：客户端收到FIN后，发送ACK报文，将ISN(s) + 1作为序列号。此时客户端处于TIME_WAIT状态。经过2MSL时间后，自动进入CLOSED状态，客户端关闭连接。

服务器收到ACK报文后，进入CLOSED状态，服务器关闭连接。

MSL：报文最大生存时间，即任何报文在网络上存在的最长时间，超出该时间报文会被丢弃。

为什么需要四次挥手

关闭连接时，客户端向服务器发送FIN仅代表客户端不再发送数据，却仍能接收数据。服务器收到FIN报文时，先回应一个ACK，但服务器可能还有数据需要处理，只有其不再发送数据时才发送FIN表示同意关闭连接。

服务器通常需要等待完成数据的发送和处理，因此其ACK和FIN会分开发送，于是比三次握手多了一次。

13.浏览器访问服务器的过程

（1）浏览器输入域名；

（2）浏览器访问DNS服务器，把域名解析为IP地址后返回给浏览器；

（3）浏览器从IP中解析出端口号；

（4）浏览器通过解析得到的IP和端口号与Web服务器建立一条TCP通道连接；

（5）建立TCP连接后，浏览器向Web服务器发送一条HTTP请求报文；

（6）Web服务器响应，并读取浏览器的请求信息，然后返回一条HTTP响应报文；

（7）Web服务器关闭HTTP连接，关闭TCP连接，显示访问的网站内容至屏幕。

14.HTTP的请求结构

客户端请求信息：请求行+请求头部+空行+请求数据。

服务器响应信息：状态行+消息报头+空行+响应正文。

15.TCP如何保证数据的可靠传输

（1）校验和

数据传输过程中，发送的数据段都会当成一个16位的整数，将整数加起来得到校验和。发送方在发送数据前计算校验和，接收方收到数据后对数据使用同样方式计算求出校验和，与发送方的进行比对。

（2）确认应答和序列号

TCP传输时对每个字节的数据进行编号，即序列号。同时每次接收方收到数据后，都会对传输方发送ACK确认应答。

（3）超时重传

发送数据后等待一段时间没有收到ACK报文，将对刚才发送的数据进行重新发送。

（4）连接管理

三次握手和四次挥手。

（5）流量控制

TCP根据接收端对数据的处理能力，决定发送方的发送速度。

（6）拥塞控制

TCP通过维护拥塞窗口实现拥塞控制，窗口会根据当前拥塞情况增大或缩小。

16.Cookie和Session的区别

Cookie：Web服务器发送给浏览器的一段信息，浏览器在本地文件中为每个Web服务器存储Cookie。之后浏览器再给特定Web服务器发送请求时，会同时发送所有为该服务器存储的Cookie。

Session：存储在Web服务器的一段信息。Session对象存储特定用户会话所需的属性及配置信息。当用户在应用程序的Web页间跳转时，存储在Session对象中的变量将不会丢失，而是一直存在于整个会话过程中。

（1）Session能够存储任意Java对象，Cookie只能存储String类型的对象。

（2）Session存放在服务器上，Cookie存放在浏览器上。

（3）Session会在一定时间内保存在服务器上，容易占用服务器性能，因此Cookie更适合在优化性能时使用。

（4）Session比Cookie更安全。

（5）Session保存数据无浏览器限制，Cookie保存的数据不能超过4K。
